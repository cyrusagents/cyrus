# Internal Changelog

This changelog documents internal development changes, refactors, tooling updates, and other non-user-facing modifications.

## [Unreleased]

### Added
- **Context-cost levers — early auto-compaction knob + read-discipline addendum.** Two of the LLM cost-optimization levers landed (the analysis's thesis: on long sessions the re-sent accumulated conversation, not intelligence, dominates spend). **Lever #1 (the dollar win):** a new optional top-level `EdgeWorkerConfig.claudeAutoCompactWindow` (`z.number().int().positive().optional()`) is forwarded all the way to the Claude Agent SDK's `Settings.autoCompactWindow` (verified settable: `Options.settings` accepts a `Settings` object, which `ClaudeRunner` already populates for `autoMemoryDirectory`). The SDK auto-compacts only near the *model's* full context window — for the 1M-token `opus-4-8[1m]` that means a session can plateau at ~200k of re-read history before ever compacting — so a smaller window forces compaction early and caps the per-turn context tax. Plumbing: `config-schemas.ts` field → `SessionOrchestrator.buildAgentRunnerConfig` reads `config.claudeAutoCompactWindow` off the already-fetched `getConfig()` and passes `autoCompactWindow` into `IssueRunnerConfigInput` → `RunnerConfigBuilder.buildIssueConfig` sets `config.autoCompactWindow` **claude-only** (Cursor manages its own context) → `ClaudeRunnerConfig.autoCompactWindow` → `ClaudeRunner` folds it into the single `sdkSettings` object alongside `autoMemoryDirectory` (previously a per-field `settings` spread that would have clobbered a second field) and projects it into the sanitized query-options diagnostics. No `ConfigManager` whitelist/`globalKeys` edits were needed — Phase A's `reconcile` is schema-driven (merge + change-detection both iterate `EdgeConfigSchema.shape`), so the Zod field alone suffices; the number field needs no path normalization. Default unset ⇒ behavior unchanged. **Lever #3 (marginal, by design):** new `contextDisciplinePromptAddendum.ts` (mirrors the `failureMode`/`browserUse`/`cloudRuntime` addendum-module pattern — constant + `append…Addendum` helper + dedicated test) wired into the `RunnerConfigBuilder` `appendSystemPrompt` composition. A terse nudge to reuse already-read files, prefer targeted reads, not re-read after edits, and offer to split oversized work into scoped issues (folds in lever #2's decomposition as a soft option rather than a behavior-changing directive). Kept deliberately short: every token is paid on every turn, and the analysis's own 93–97% cache-hit rate means re-reads are mostly cache-served, so read-discipline is a small slice — the volume of accumulated context (lever #1's target) is the real driver. **Lever #4 (model routing off Opus for mechanical turns) is intentionally NOT implemented:** the SDK fixes the model once per session at `query()` construction (`ClaudeRunner.ts` `options.model`), so per-turn model swaps are architecturally impossible; routing a mechanical turn to a cheaper model would require running it as a separate session (the same session-split rewrite lever #1 sidesteps), and lever #1 addresses the underlying context-tax cost directly. Tests: `contextDisciplinePromptAddendum.test.ts`, `RunnerConfigBuilder.autoCompactWindow.test.ts` (claude sets / unset-passthrough / cursor-omits). ([#18](https://github.com/TropicalDog17/cyrus/pull/18))
- **Langfuse traces now carry a Cyrus build `version` for cross-deploy comparison.** The exporter stamped no version, so every session looked like the same build and cost/latency couldn't be compared before vs. after a change. A new `resolveTraceVersion(env)` composes `<semver>+<commit>` and is set as the trace `version` (the field Langfuse's version-comparison view keys on) plus mirrored into trace `metadata.version`. Sources with graceful fallback: semver from `CYRUS_VERSION` env → runner `package.json` version (read once, lazily, via `import.meta.url` → `../package.json`, degrading to `"unknown"` rather than throwing) → and the short commit from `CYRUS_BUILD_COMMIT` (omitted from the string when absent, e.g. local dev). Semver alone is static across a change (all packages share one version), so the appended commit is what makes each deploy distinct without needing a version bump. `scripts/deploy-local.sh` now reads the `cyrus-ai` package version and injects both `Environment=CYRUS_VERSION=<semver>` and `Environment=CYRUS_BUILD_COMMIT=<short-sha>` into the `cyrus.service` systemd drop-in (and logs `version <semver>+<commit>` on restart), so the live agent's traces are attributable to an exact build. Tests: `resolveTraceVersion` env-combination matrix + an export-level assertion that the trace body and its metadata carry `<semver>+<commit>`. ([#19](https://github.com/TropicalDog17/cyrus/pull/19))
- **Deploy-on-merge for the self-hosted "troppc" agent.** Added `.github/workflows/deploy.yml` (self-hosted runner, label `cyrus-deploy`) + `scripts/deploy-local.sh`: every push to `main` fetches the branch tip into a dedicated deploy clone (`~/cyrus-deploy`, kept separate from the dev working copy so uncommitted work is never hard-reset), runs `pnpm install --frozen-lockfile && pnpm build`, stamps `BUILD_INFO.txt`, repoints the `cyrus.service` systemd user unit at the freshly-built entrypoint via an idempotent drop-in, and restarts it. The script is also runnable by hand (`CYRUS_DEPLOY_SKIP_RESTART=1` for a build-only dry run). Requires a self-hosted runner registered to the repo with the `cyrus-deploy` label.

### Fixed
- **Langfuse export no longer multiplies token cost per turn.** Claude Code writes a single assistant message to the transcript as several JSONL records (one per content block — thinking / text / each tool_use), all repeating the same cumulative `message.usage`. The exporter keyed each generation off the per-record `uuid`, so one turn emitted N generations each stamped with the full token usage, inflating a trace's reported cost by the number of blocks (observed ~2.4–3× on real sessions; a $12 session read as $37). Generations are now aggregated and keyed by assistant `message.id` — exactly one per turn — with tool spans still emitted per `tool_use`. Additionally, input is now reported via `usageDetails` broken down into `input` / `cache_read_input_tokens` / `cache_creation_input_tokens` (previously all folded into one flat `input` figure), so Langfuse can price cache reads/writes at their cheaper rates instead of charging every input token at the full uncached rate.
- **Langfuse spans now carry real per-turn / per-tool durations instead of zero.** The exporter stamped every generation and tool span with `endTime === startTime`, so each turn showed as an instantaneous point — Langfuse could report a session's total wall-clock but not which turns burned it (the exact gap that blocked latency analysis of high-context sessions). Durations are now reconstructed from the transcript record timestamps already present in the JSONL: a turn runs from the record that triggered it (the user prompt or the prior `tool_result`) to the assistant record that answered it, and a tool span runs from that response until its matching `tool_result` lands. A `clampEnd` guard keeps an observation from ever ending before it starts (clock skew / out-of-order records collapse to a zero-duration point). `timeToFirstToken` is deliberately left unset — a post-hoc transcript records no streaming timing, so a reported `0` would be a fabrication rather than a measurement.
